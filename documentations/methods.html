Backend Method Definitions

1. Member Service

Method Parameters Returns Logic Description
registerMember(memberData) {firstName, lastName, address, phone, email, membershipType} memberId Insert into Member. Check email uniqueness. Set membership_date = today. Default membership_type = 'regular'. Return new member_id.
getMemberProfile(memberId) memberId Member object Select from Member where member_id = ?. Include current loans count and total unpaid fines.
updateMember(memberId, updatedData) memberId, updatedFields boolean Update allowed fields (address, phone, email). Enforce email uniqueness.
deactivateMember(memberId) memberId boolean Only if no active loans and no unpaid fines. Soft delete (add active flag) or physically remove if allowed.
getMemberLoans(memberId, includeHistory) memberId, includeHistory (bool) list of Loan objects If includeHistory=false, return only open loans (return_date IS NULL). Else return all loans for that member.
getMemberFines(memberId, unpaidOnly) memberId, unpaidOnly list of Fine objects Join Loan and Fine. If unpaidOnly=true, filter where paid_status = FALSE.
payFine(fineId, amountPaid) fineId, amountPaid boolean Update paid_status = TRUE if amountPaid >= fine.amount. Optionally record payment date.

Tips for Member Service:

  • Hash passwords if adding authentication (not in current schema; extend with password_hash).
  • Use database transactions when updating member data and related fines.
  • Index email for quick lookup.

2. Catalog (Book / Author / Category) Service

Method Parameters Returns Logic Description
addBook(bookData, authorIds, categoryIds) {isbn, title, edition, publicationYear, publisherId, description}, authorIds[], categoryIds[] bookId Insert into Book. Then insert into Book_Author for each authorId. Insert into Book_Category for each categoryId.
updateBook(bookId, updatedBookData) bookId, updatedFields boolean Update Book table. Changing authors/categories requires separate methods.
deleteBook(bookId) bookId boolean Only if no copies exist (i.e., SELECT COUNT(*) FROM Book_Copy WHERE book_id = ? = 0). Otherwise return error.
searchBooks(searchCriteria) {title, author, isbn, category, publicationYear} list of Book objects Dynamic SQL. Use LIKE for partial title/author name. Join with Book_Author and Author for author search. Join with Book_Category and Category for category.
getBookDetails(bookId) bookId Book object with authors, categories, and total copy count per branch SELECT from Book + aggregated subqueries.
addAuthor(authorData) {firstName, lastName, biography} authorId Insert into Author. Return new ID.
listAllAuthors() none list of Author objects Simple SELECT.
addCategory(categoryName) name categoryId Insert into Category. Check for duplicate name.
getBooksByCategory(categoryId) categoryId list of Book objects Join Book_Category + Book.

Tips for Catalog Service:

  • Implement pagination (offset, limit) for searchBooks and listAllAuthors.
  • Use full‑text indexes on title, description for faster search.
  • Validate ISBN format (e.g., ISBN‑10 or ISBN‑13) before insertion.

3. Inventory (Book Copy & Branch) Service

Method Parameters Returns Logic Description
addBookCopy(bookId, branchId, acquisitionDate) bookId, branchId, acquisitionDate copyId Insert into Book_Copy with status = 'available'. Generate new copy_id.
updateCopyStatus(copyId, newStatus) copyId, newStatus boolean Update status in Book_Copy. Allowed transitions: available ↔ loaned, available ↔ reserved, loaned → available (when returned), available → lost, etc.
getAvailableCopies(bookId, branchId) bookId, branchId (optional) list of copyId SELECT copy_id FROM Book_Copy WHERE book_id = ? AND status = 'available' [AND branch_id = ?].
transferCopy(copyId, targetBranchId) copyId, targetBranchId boolean Update branch_id of that copy. Ensure copy is not currently loaned.
removeCopy(copyId) copyId boolean Only if status = 'lost' or 'available' with no active loan. Delete or mark as discarded (add discarded flag).
addBranch(branchData) {name, location, phone} branchId Insert into Library_Branch.
getBranchInventory(branchId) branchId list of {bookTitle, copyId, status} Join Book_Copy + Book.

Tips for Inventory Service:

  • Use optimistic locking or row‑level locks when updating status to prevent double‑loaning (e.g., SELECT FOR UPDATE).
  • Create a stored procedure to atomically change status from ‘available’ to ‘loaned’ when issuing a book.

4. Circulation (Loan & Return) Service

Method Parameters Returns Logic Description
loanBook(copyId, memberId, loanDate, dueDate) copyId, memberId, loanDate (default today), dueDate (calculated) loanId 1. Check copy status = ‘available’. 2. Check member has no overdue fines > $10 (application rule). 3. Insert into Loan. 4. Update Book_Copy.status = ‘loaned’. 5. If there is a reservation for this book by another member, optionally create waiting list.
returnBook(loanId, returnDate) loanId, returnDate (default today) fineAmount (0 if no fine) 1. Get Loan record. 2. Update return_date = returnDate. 3. Compute overdue days: returnDate > due_date ? days late : 0. 4. If overdue, call calculateAndCreateFine(loanId, overdueDays). 5. Update Book_Copy.status = ‘available’. 6. If there are pending reservations for that book, change status to ‘reserved’ and notify next member.
calculateAndCreateFine(loanId, overdueDays) loanId, overdueDays fineId Insert into Fine with amount = overdueDays * dailyFineRate (e.g., 0.50 per day). paid_status = FALSE.
renewLoan(loanId, newDueDate) loanId, newDueDate boolean Allowed if no reservation pending on that book and not exceeding max renewals (e.g., 2). Update due_date = newDueDate.
getOverdueLoans(asOfDate) asOfDate list of Loan objects SELECT FROM Loan WHERE return_date IS NULL AND due_date < asOfDate.
getLoanHistory(copyId) copyId list of Loan objects All loans for a specific physical copy.

Tips for Circulation Service:

  • Use database transactions for loan and return: multiple updates (Loan insert + Book_Copy update) must be atomic.
  • Implement a queue or scheduler to automatically calculate fines daily (batch job).
  • Store dailyFineRate in a Settings table instead of hardcoding.
  • Use DATETIME for loan_date/due_date if you need hour precision.

5. Reservation Service

Method Parameters Returns Logic Description
placeReservation(bookId, memberId, branchId, expirationDays) bookId, memberId, branchId, expirationDays (e.g., 7) reservationId 1. Check member has no active reservations exceeding limit (e.g., 5). 2. Insert into Reservation with status='pending', expiration_date = today + expirationDays. 3. If any copy is available now, optionally auto‑fulfill.
cancelReservation(reservationId) reservationId boolean Update status = 'cancelled'.
fulfillReservation(reservationId, copyId) reservationId, copyId boolean 1. Change Reservation.status = ‘fulfilled’. 2. Optionally create a loan for that member using that copy (or mark copy as ‘reserved’ until member picks up). 3. Notify member.
getPendingReservations(bookId) bookId list of Reservation Used to determine who gets the next available copy (ordered by reservation_date).
expireOldReservations(currentDate) currentDate count expired Batch job: update status = 'expired' where expiration_date < currentDate AND status='pending'.

Tips for Reservation Service:

  • Use a FIFO queue based on reservation_date for each book.
  • When a copy is returned, automatically check pending reservations and either notify or auto‑loan.
  • Consider a “pickup window” (e.g., 48 hours) after fulfillment before moving to next reservation.

6. Fine & Payment Service

Method Parameters Returns Logic Description
getMemberTotalUnpaidFine(memberId) memberId total amount SUM of amount from Fine JOIN Loan WHERE Loan.member_id = ? AND Fine.paid_status = FALSE.
payFine(fineId, paymentAmount) fineId, paymentAmount boolean If paymentAmount >= fine amount, set paid_status=TRUE and optionally store payment_date. If partial payment allowed (depends on policy), reduce amount and keep as unpaid.
waiveFine(fineId, reason) fineId, reason boolean Administrative override: set paid_status=TRUE and record waiver reason.
getAllUnpaidFines(asOfDate) asOfDate list of Fine objects For reporting / batch reminders.

Tips for Fine Service:

  • Use a decimal(6,2) for currency; ensure rounding to nearest cent.
  • Avoid floating point errors: store in cents (integer) if preferred.
  • Add payment_transaction_id to track payments via external gateway.

Method Parameters Returns Logic Description

authenticateStaff(staffId, password) staffId, password boolean Requires adding password_hash column to Staff (not in base schema).

  • Implement role‑based access control (e.g., role = ‘admin’, ‘librarian’, ‘assistant’).

Implementation Tips for All Methods (Backend)

A. Transaction Management

Always use database transactions for operations that update multiple tables:

  • loanBook (insert Loan + update Book_Copy)
  • returnBook (update Loan + update Book_Copy + insert Fine if overdue)
  • placeReservation (insert Reservation + optionally update Book_Copy status to ‘reserved’)
  • payFine (update Fine + potentially log payment)

Example (Python/Psycopg2):

with connection: with connection.cursor() as cur: cur.execute("UPDATE book_copy SET status = 'loaned' WHERE copy_id = %s", (copy_id,)) cur.execute("INSERT INTO loan (copy_id, member_id, loan_date, due_date) VALUES (%s,%s,%s,%s)", (copy_id, member_id, loan_date, due_date))

B. Validation & Business Rules

  • Validate input parameters before touching the database (e.g., ISBN format, date ranges, non‑negative amounts).
  • Enforce constraints like “member cannot borrow if unpaid fines > $10” at the service layer, not just DB.
  • Use SQL constraints (CHECK, FOREIGN KEY, UNIQUE) as a second line of defense.

C. Performance Optimization

  • Create indexes on:
    • Loan(member_id, return_date) – for active loans queries.
    • Loan(copy_id, loan_date) – for history.
    • Book_Copy(book_id, branch_id, status) – for availability checks.
    • Reservation(book_id, status, reservation_date) – for queue processing.
    • Fine(loan_id) and Fine(paid_status).
  • Use connection pooling (e.g., HikariCP, PgBouncer) to handle concurrent users.
  • Batch operations: e.g., expireOldReservations should update many rows with one SQL statement.

D. Error Handling & Logging

  • Define custom exceptions: BookNotAvailableError, MemberBlockedError, FineMismatchError.
  • Log every state change (loan, return, payment) to an audit table for legal/financial traceability.

E. API Design (if building REST endpoints)

  • Use standard HTTP methods and status codes:
    • POST /api/members → registerMember
    • POST /api/loans → loanBook
    • PUT /api/loans/{loanId}/return → returnBook
    • GET /api/books?title=... → searchBooks
    • POST /api/reservations → placeReservation
  • Document with OpenAPI / Swagger.

F. Testing Strategy

  • Write unit tests for each service method (mocking the DB).
  • Write integration tests using a test database (e.g., H2, SQLite, or PostgreSQL with transactions rolled back).
  • Test edge cases: double loan of same copy, reservation expiry, fine calculation at DST boundaries.

G. Security

  • Never expose internal IDs directly in URLs without authorization checks (e.g., member can only access their own loans).
  • Use parameterized queries or ORM to prevent SQL injection.
  • Rate limit public endpoints (e.g., search) to prevent abuse.

H. Development Workflow Suggestion

  1. Start with Member and Catalog services (read‑only).
  2. Add Inventory (add copies, view available).
  3. Implement Circulation (loan/return) – the core.
  4. Add Reservation and Fine services.
  5. Build admin Reports.
  6. Write integration tests for each major user journey.
